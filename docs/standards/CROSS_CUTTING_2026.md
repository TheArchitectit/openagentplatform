# 2026 Cross-Cutting Guardrails — Universal Standards

> Security and quality standards that apply to ALL language profiles, regardless of stack.

**Version:** 2.0.0
**Effective:** 2026-Q2
**Applies To:** All Sentinel language profiles (Go, Python, TS, Rust, Java, Kotlin, C#, C++, PHP, Ruby, Swift, Scala, R, SQL, GDScript/Godot, Docker/DevOps, AI/ML)

## 1. Supply Chain Security (SBOM / SLSA)

Every project must implement supply chain integrity. The 2024-2026 attacks on npm, PyPI, and npm packages (xz/liblzma, pytest-fixture, etc.) make this non-negotiable.

### 1.1 Software Bill of Materials (SBOM)

**Rule:** Every build must produce an SBOM.

| Ecosystem | Tool | Format |
|-----------|------|--------|
| Go | `go version -m` / `syft` | SPDX/CycloneDX |
| Python | `pip-audit` / `cyclonedx-bom` | CycloneDX |
| Node/TS | `npm sbom` / `syft` | SPDX/CycloneDX |
| Rust | `cargo-cyclonedx` | CycloneDX |
| Java | `cyclonedx-maven-plugin` | CycloneDX |
| Docker | `syft <image>` | SPDX/CycloneDX |
| Godot | Manual manifest in `project.godot` | Custom |

**Enforcement:**
- On `build` command, Sentinel checks for SBOM generation step.
- If CI pipeline exists, SBOM must be generated as an artifact.
- If SBOM is missing, Sentinel issues a warning.
- **Action:** Warning for missing SBOM. Block for production releases without SBOM.

### 1.2 SLSA Provenance

**Rule:** Build artifacts must have SLSA provenance.

**Enforcement:**
- Level 1 (Provenance exists): CI generates signed provenance for builds.
- Level 2 (Hosted build): Builds run on CI, not developer machines.
- Level 3 (Non-falsifiable): Build platform is hardened.
- Sentinel checks for `slsa-framework/slsa-github-generator` or equivalent.
- **Action:** Info for missing L1. Warning for release builds without provenance.

### 1.3 Dependency Verification

**Rule:** All dependencies must be verified before use.

| Ecosystem | Mechanism |
|-----------|-----------|
| Go | `go.sum` checksums (mandatory) |
| Python | `pip --require-hashes` or `hashin` |
| Node/TS | `package-lock.json` + `npm audit signatures` |
| Rust | `Cargo.lock` + `cargo-audit` |
| Java | Gradle dependency verification (`verification-metadata.xml`) |
| Docker | `cosign verify` for signed images |

**Enforcement:**
- Sentinel checks for lock files before any `install` operation.
- If lock file is missing or stale, Sentinel blocks the build.
- **Action:** BLOCK for missing lock files.

## 2. Secret Scanning

### 2.1 Pre-Commit Secret Detection

**Rule:** No secrets, tokens, or credentials in source code.

**Tools to integrate:**

| Tool | Languages | Type |
|------|-----------|------|
| `gitleaks` | All | Pre-commit hook |
| `trufflehog` | All | Git history scanner |
| `detect-secrets` | Python/JS | Baseline scanner |
| `GitHub Secret Scanning` | All | Push protection |

**Enforcement:**
- Sentinel installs `gitleaks` pre-commit hook on `sentinel init`.
- On `validate_file_edit`, Sentinel checks for secret patterns.
- Pattern: API keys (AKIA*, sk-*, AIza*), tokens (ghp_*, gho_*), private keys (-----BEGIN RSA).
- **Action:** CRITICAL BLOCK for any detected secret.

### 2.2 Secret Rotation Guard

**Rule:** Committed secrets must be rotated immediately.

**Enforcement:**
- If `gitleaks` finds a secret in git history (not just current diff):
  1. CRITICAL alert issued.
  2. Agent is told to rotate the credential immediately.
  3. `git filter-repo` or BFG recommended for history cleanup.
- **Action:** Critical alert + rotation instructions.

## 3. AI Code Generation Awareness

AI-generated code (Copilot, Claude, ChatGPT, etc.) introduces unique risks. These guardrails apply when code is produced or assisted by AI agents.

### 3.1 The "AI-Assisted" Annotation

**Rule:** Code substantially generated by AI should be annotated.

**Enforcement:**
- Sentinel checks git commits for co-author patterns.
- Recommended: Add `Co-authored-by: AI Assistant <ai@sentinel>` to commits with AI-generated code.
- This aids in later review and attribution.
- **Action:** Info notice if no AI annotation detected.

### 3.2 The AI Output Verification Guard

**Rule:** AI-generated code must be reviewed before merge.

**Enforcement:**
- If commit is marked as AI-assisted:
  - Code review is MANDATORY (no direct push to main).
  - Test coverage must be ≥80% for AI-generated functions.
  - No AI-generated code in security-critical paths without manual audit.
- **Action:** Block direct merge of AI-generated code to main branch.

### 3.3 The Hallucinated Dependency Guard

**Rule:** AI-generated import statements must reference real packages.

**Enforcement:**
- Sentinel verifies every `import` / `require` / `use` against the package registry.
- AI models frequently hallucinate packages that don't exist (or worse, typosquatted lookalikes).
- Check: Package exists on registry (npm, PyPI, crates.io, pkg.go.dev).
- Check: Package has > 100 downloads and > 1 maintainer.
- **Action:** BLOCK for non-existent packages. Warning for low-trust packages.

### 3.4 The AI Logic Verification Guard

**Rule:** AI-generated algorithms must be tested against known correct outputs.

**Enforcement:**
- For mathematical functions, crypto operations, or business logic:
  - Sentinel requires at least 3 test cases with known correct outputs.
  - If only "happy path" tests exist for AI-generated code, warning.
- **Action:** Warning for untested AI-generated logic.

## 4. License Compliance

### 4.1 License Compatibility Check

**Rule:** All dependencies must have compatible licenses.

**Enforcement:**
- Sentinel scans dependency licenses on `install` and `build`.
- Incompatible combinations flagged:
  - GPL in non-GPL project → CRITICAL
  - AGPL in any SaaS project → CRITICAL
  - SSPL/BSL in open-source project → WARNING
  - Unlicense/CC0 → INFO (verify intent)
- Tools: `go-licenses`, `pip-licenses`, `license-checker` (npm), `cargo-lichking`
- **Action:** BLOCK for license incompatibilities.

### 4.2 License File Mandate

**Rule:** Every project must include a LICENSE file.

**Enforcement:**
- Sentinel checks for LICENSE / LICENSE.md / COPYING on init.
- If missing, warning issued with recommendation.
- **Action:** Warning for missing license file.

## 5. Container Security (Applies When Docker/Podman is Used)

### 5.1 Image Vulnerability Scanning

**Rule:** Container images must be scanned for CVEs before deployment.

**Tools:** `trivy`, `grype`, `docker scout`

**Enforcement:**
- Sentinel runs vulnerability scan on `build` if Dockerfile detected.
- Critical/High CVEs → BLOCK.
- Medium CVEs → Warning.
- Low CVEs → Info.
- **Action:** BLOCK for Critical/High CVEs in production images.

### 5.2 Multi-Stage Build Mandate

**Rule:** Production images must use multi-stage builds.

**Enforcement:**
- Sentinel scans Dockerfile for multi-stage pattern (multiple FROM instructions).
- If single-stage build for production, warning.
- Build tools, dev dependencies, and source code should not be in final image.
- **Action:** Warning for single-stage production Dockerfiles.

## 6. OWASP Top 10 (2025/2026) — Universal Checks

These apply regardless of language:

| OWASP Category | Sentinel Check | Action |
|----------------|---------------|--------|
| A01: Broken Access Control | Route auth decorators, RBAC enforcement | BLOCK if missing |
| A02: Cryptographic Failures | Hardcoded secrets, weak crypto (MD5, SHA1 for passwords) | BLOCK |
| A03: Injection | SQL injection patterns, command injection, XSS | BLOCK |
| A04: Insecure Design | Missing rate limiting, no input validation | WARN |
| A05: Security Misconfiguration | Debug mode in prod, default credentials, CORS * | BLOCK |
| A06: Vulnerable Components | Known CVEs in dependencies (npm audit, pip-audit, cargo-audit) | BLOCK for Critical |
| A07: Auth Failures | Weak password policies, no MFA, session fixation | WARN |
| A08: Data Integrity Failures | Unsigned CI pipelines, unverified dependencies | WARN |
| A09: Logging Failures | Missing audit logging, sensitive data in logs | WARN |
| A10: SSRF | Unvalidated URL fetches, internal network access from user input | BLOCK |

## 7. Mobile/Game-Specific Cross-Cutting Rules

When building for mobile platforms (Android/iOS) or game engines:

### 7.1 Performance Budgets

| Metric | Mobile Limit | Desktop Limit |
|--------|-------------|---------------|
| APK/IPA size | 100 MB | 500 MB |
| Startup time | 3 seconds | 5 seconds |
| Memory usage | 512 MB | 2 GB |
| Texture resolution | 2048x2048 | 4096x4096 |
| Frame time | 16.67ms (60fps) | 8.33ms (120fps) |

### 7.2 Privacy Compliance (GDPR/CCPA)

- No location access without explicit consent
- No identifier collection without privacy policy
- Data export and deletion capabilities
- Age-appropriate design for games targeting minors

### 7.3 App Store Compliance

- No undocumented API usage
- No dynamic code execution (eval, exec) in iOS apps
- Android: Target API 34+ for Play Store
- Proper app signing and keystore management
